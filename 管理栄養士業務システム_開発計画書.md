## 1. プロジェクト概要

### 1.1 目的

管理栄養士業務（献立作成・栄養計算・発注集計・帳票出力）を一元化し、以下を実現する：

- **転記作業の削減**
- **計算精度向上**
- **業務効率化**

### 1.2 開発方針

| 方針 | 内容 |
|------|------|
| **段階的技術進化** | Small → Strong。フェーズごとに技術を進化させる |
| **データモデル中心設計** | 食材・料理・レシピの3層構造を核とする |
| **データ互換維持** | フェーズ移行時もデータ互換を維持する |
| **動く成果物優先** | 常に「動く成果物」を優先する |

## 2. フェーズ構成（確定）

### Phase1：Python + CSV（MVP）

| 項目 | 内容 |
|------|------|
| **目的** | 最短で業務フローを一気通貫で動かす |
| **技術** | Python（CLI）、CSV保存 |
| **成果物** | 献立登録CLI、栄養価計算機能、献立表Excel出力、発注集計機能、CSVスキーマ定義 |
| **完了条件** | 1日分の献立登録→栄養計算→Excel出力→発注集計が連続実行可能。テストデータで計算誤差±許容範囲内。エラー処理が仕様通り動作 |

### Phase2：Go + SQLite（型安全・単体バイナリ化）

| 項目 | 内容 |
|------|------|
| **目的** | データ整合性強化と配布性向上 |
| **技術** | Go（CLI）、SQLite |
| **成果物** | SQLiteスキーマ、CSV→SQLite移行ツール、単体バイナリ実行ファイル、型安全な栄養計算ロジック |
| **完了条件** | Phase1のCSVデータが完全移行可能。入出力結果がPhase1と一致。主要ロジックに単体テスト導入 |

### Phase3：Go API + TypeScript + React（Web UI）

| 項目 | 内容 |
|------|------|
| **目的** | 操作性向上と現場導入準備 |
| **技術** | Go（REST API）、TypeScript + React |
| **成果物** | 献立入力画面、栄養計算表示画面、発注集計ダウンロード、API仕様書 |
| **完了条件** | Web上で献立登録〜帳票出力まで実行可能。APIとフロントが分離構造。同一ロジックを再利用 |

### Phase4：PostgreSQL + 本格運用

| 項目 | 内容 |
|------|------|
| **目的** | 複数施設対応・本番運用体制確立 |
| **技術** | Go（原則）、PostgreSQL、React |
| **成果物** | マルチ施設設計、権限管理、監査ログ、バックアップ設計、運用Runbook |
| **完了条件** | 複数施設データ分離。定期バックアップ自動化。運用ドキュメント完成 |

## 3. データ互換ポリシー（重要）

### 3.1 データの正（Source of Truth）

常に以下の論理モデルを正とする：

- **ingredients**（食材マスタ）
- **dishes**（献立候補）
- **recipe**（レシピ）
- **menus**（献立）

### 3.2 schema_version

すべてのデータ形式に `schema_version` を持たせる。

| 形式 | 格納場所 |
|------|----------|
| CSV | data/schema.json |
| SQLite | version テーブル |

### 3.3 移行原則

- CSVは常にインポート可能であること
- 変換ツールは破壊的変更をしないこと
- 同一入力→同一出力が再現されること
