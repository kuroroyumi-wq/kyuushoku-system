# 管理栄養士業務システム アーキテクチャ設計書

**版数**：2.0  
**作成日**：2026年3月3日  
**方針**：段階的技術進化（Small → Strong）、データモデル中心設計

---

## 1. 設計方針

### 1.1 開発計画書v2.0との整合

本アーキテクチャは**開発計画書v2.0**に準拠する。

- 段階的技術進化
- データモデル中心設計（食材・料理・レシピ3層構造）
- フェーズ移行時もデータ互換を維持
- 常に「動く成果物」を優先

### 1.2 全体構成（目標状態）

```
┌─────────────────────────────────────────────────────────────┐
│                      クライアント層                           │

         │                 │
         ▼                 ▼

         ▼
```

## 2. フェーズ別技術スタック

| フェーズ | データ | バックエンド/CLI | フロントエンド |
|----------|--------|------------------|----------------|
| **Phase1** | CSV | Python | なし |
| **Phase2** | SQLite | Go | なし |
| **Phase3** | SQLite | Go + REST API | TypeScript + React |
| **Phase4** | PostgreSQL | Go（原則） | React |

**Rust**：高負荷バッチ処理用途のみ検討。技術選定は「保守性＞流行」。

## 3. リポジトリ構成（monorepo）

```
project-root/
├── data/              # 共通データ
│   ├── schema.json    # schema_version
│   ├── schema.sql     # SQLite スキーマ（Phase2）
│   ├── ingredients.csv
│   ├── dishes.csv
│   ├── recipe.csv
│   ├── menus.csv
│   └── menu.db        # SQLite DB（移行後）
├── docs/
├── tools/             # 移行ツール
│   └── migrate_csv_to_sqlite.py
├── phase1_python/     # Phase1実装
├── backend_go/        # Phase2（Go CLI + SQLite）
└── frontend_web/      # Phase3以降
```

---

## 4. データ互換ポリシー

### 4.1 schema_version

- **CSV**：data/schema.json で管理
- **SQLite**：version テーブルで管理

### 4.2 移行原則

- CSVは常にインポート可能
- 変換ツールは破壊的変更をしない
- 同一入力→同一出力が再現される
